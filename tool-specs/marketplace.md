# verusidx-marketplace-mcp — Tool Specs

On-chain offers and trades — identity swaps, currency offers, and decentralized atomic swaps.

Every RPC tool requires a `chain` parameter — no default, no auto-selection. The shared library routes the RPC call to the correct daemon by looking up host:port in the chain registry.

**Read-only mode (`VERUSIDX_READ_ONLY=true`):** `makeoffer`, `takeoffer`, and `closeoffers` are not registered. Only `getoffers` and `listopenoffers` remain available. Read-only mode is per-MCP: marketplace-mcp can be read-only while send-mcp is read-write, or vice versa.

---

## 1. `getoffers`

**Description:**
Get all open offers for a specific currency or identity. Returns both buy and sell offers — offers *of* the asset and offers *for* the asset. Use `iscurrency` to switch between currency offers and identity offers.

The response keys are dynamic and generated by the daemon based on the queried asset (e.g., `currency_{id}_offers_in_currency_{id}`, `ids_for_currency_{id}`, `currency_{id}_for_ids`, `ids_for_id_{id}`). Parse response keys dynamically — they are not fixed strings.

Use this to discover what's available on the market before making or taking offers.

**Input Schema:**

| Param | Type | Required | Description |
|---|---|---|---|
| `chain` | string | Yes | Chain to query (e.g., `"VRSC"`, `"vrsctest"`) |
| `currencyorid` | string | Yes | Currency name/i-address or identity name/i-address to check for offers |
| `iscurrency` | boolean | No | If `true`, look for currency offers. If `false` (default), look for identity offers. |
| `withtx` | boolean | No | If `true`, include serialized hex of the exchange transaction for signing. Default: `false`. |

**Annotations:**
```json
{
  "readOnlyHint": true,
  "destructiveHint": false,
  "idempotentHint": true,
  "openWorldHint": false
}
```

**Output:**

Returns the daemon's `getoffers` response directly. The top-level object has one or more dynamically-named keys containing arrays of offer objects.

**Offer object fields (currency offers):**

| Field | Type | Description |
|---|---|---|
| `currencyid` | string | i-address of the currency being offered or requested |
| `price` | number | Price per unit |
| `offer` | object | Offer details (see below) |

**Offer object fields (identity offers):**

| Field | Type | Description |
|---|---|---|
| `identityid` | string | i-address of the identity being offered or requested |
| `price` | number | Price (in the paired currency) |
| `offer` | object | Offer details (see below) |

**Inner offer detail fields:**

| Field | Type | Description |
|---|---|---|
| `offer` | object | What the maker is offering — currency map (`{"i-address": amount}`) and/or identity info (`{"name": "...", "identityid": "...", "systemid": "...", "original": number}`) |
| `accept` | object | What the maker wants in return — same structure as `offer` |
| `blockexpiry` | number | Block height at which this offer expires |
| `txid` | string | Transaction ID of the offer — use this in `takeoffer` |

**Key fields for agents:**
- `blockexpiry` — check against current block height. Expired offers cannot be taken.
- `txid` — pass this to `takeoffer` to accept the offer.
- The `offer` side may include a small native currency amount alongside the main offer — this is the fee embedded in the offer transaction.
- `original` in identity offers — indicates whether this is the original identity (value varies).

## 2. `listopenoffers`

**Description:**
List open offers from the current wallet. Shows what this wallet has offered on-chain. Can filter by expired/unexpired status.

Use this to check the status of offers the agent has created, or to find offers that need to be closed.

**Input Schema:**

| Param | Type | Required | Description |
|---|---|---|---|
| `chain` | string | Yes | Chain to query (e.g., `"VRSC"`, `"vrsctest"`) |
| `unexpired` | boolean | No | Include unexpired (active) offers. Default: `true`. |
| `expired` | boolean | No | Include expired offers. Default: `true`. |

**Annotations:**
```json
{
  "readOnlyHint": true,
  "destructiveHint": false,
  "idempotentHint": true,
  "openWorldHint": false
}
```

**Output:**

Returns an array of offer objects from this wallet. Each offer has:

| Field | Type | Description |
|---|---|---|
| `expires` | number | Block height at which this offer expires |
| `txid` | string | Transaction ID of the offer |
| `offer` | object | What the wallet is offering — includes full output details (identity definition, currency amounts, etc.) |
| `for` | object | What the wallet wants in return — same structure as `offer` |

The `offer` and `for` objects have a different structure than `getoffers` — they include full output details:

| Field | Type | Description |
|---|---|---|
| `type` | string | Output type (e.g., `"cryptocondition"`) |
| `identityprimary` | object | (identity offers) Full identity definition including `name`, `identityaddress`, `parent`, `systemid`, `primaryaddresses`, `minimumsignatures`, `revocationauthority`, `recoveryauthority`, etc. |
| `spendableoutput` | boolean | Whether the output is spendable |
| `nativeout` | number | Native currency amount in the output |

**Key fields for agents:**
- `expires` — compare against current block height to determine if the offer is still active.
- `txid` — use this to close the offer with `closeoffers`, or reference it when constructing a `takeoffer`.
- `identityprimary.parent` — always present in identity offers, shows the namespace.

## 3. `makeoffer`

**Description:**
Create a new on-chain atomic swap offer. Offers are fully decentralized — no intermediary, no escrow. The offer transaction locks the offered asset on-chain until the offer is taken, expires, or is closed.

Supports all 4 combinations: currency-for-currency, currency-for-identity, identity-for-currency, and identity-for-identity.

When the offer involves an identity (in `offer` or `for`), always include `parent` in the identity definition to ensure correct namespace resolution, especially for sub-identities.

**ID Control Tokens:** If an identity has a control token (check `identity.flags` in `getidentity` — `flags: 5` indicates a control token exists; verify with `getcurrency` using the ID name), the identity **cannot** be offered or requested directly. Instead, offer or request the control token as a currency: `{"currency": "idname.parent", "amount": 0.00000001}`. There is exactly 1 satoshi (0.00000001) of each control token. Whoever holds the control token has revoke/recover authority over the identity. The daemon will reject attempts to use the identity directly in makeoffer when a control token exists.

**Disabled in read-only mode.**

**Input Schema:**

| Param | Type | Required | Description |
|---|---|---|---|
| `chain` | string | Yes | Chain to transact on (e.g., `"VRSC"`, `"vrsctest"`) |
| `fromaddress` | string | Yes | VerusID or wildcard address to send funds from. `"*"`, `"R*"`, or `"i*"` are valid wildcards. |
| `offer` | object | Yes | What to offer — see Offer/For Object below |
| `changeaddress` | string | Yes | Transparent address or VerusID (`"ID@"`) for change. |
| `expiryheight` | number | No | Block height at which this offer expires. Default: current height + 20 (~20 minutes). Consider setting longer for offers that should remain open. |
| `for` | object | Yes | What to accept in return — see Offer/For Object below |
| `returntx` | boolean | No | If `true`, return signed transaction hex instead of broadcasting. Default: `false`. |
| `feeamount` | number | No | Custom fee amount. Default: `0.0001`. |

**Offer/For Object:**

The `offer` and `for` objects each describe one side of the swap. Each can be either a currency amount or an identity definition.

*Currency offer/request (also used for ID control tokens):*
```
{
  "currency": "currencynameorid",
  "amount": number
}
```

*Identity offer/request (only for IDs without control tokens):*
```
{
  "name": "identityname",
  "parent": "parentnameorid",
  "primaryaddresses": ["R-address", ...],
  "minimumsignatures": 1,
  "revocationauthority": "nameorID",
  "recoveryauthority": "nameorID",
  ...
}
```

When offering an identity (without a control token), the `offer` object can use shorthand: `{"identity": "idname@"}`.

When requesting an identity in `for`, provide the full identity definition (same structure as `registeridentity` / `updateidentity`). Always include `parent` to ensure correct namespace resolution.

**Annotations:**
```json
{
  "readOnlyHint": false,
  "destructiveHint": false,
  "idempotentHint": false,
  "openWorldHint": true,
  "confirmationHint": true
}
```

**Output:**

| Field | Type | Description |
|---|---|---|
| `txid` | string | Transaction ID of the offer — use this for `takeoffer` and `closeoffers` |
| `oprettxid` | string | Transaction ID of the OP_RETURN output (internal bookkeeping — agents can ignore) |
| `hex` | string | Serialized transaction hex (only if `returntx` is true) |

**Key fields for agents:**
- `txid` — save this to track the offer. Pass to `closeoffers` to cancel, or share so a counterparty can use `takeoffer`.
- `expiryheight` — defaults to ~20 minutes. For marketplace listings, set much higher (e.g., current height + 1440 for ~1 day, or + 10080 for ~1 week).
- When offering an identity, the offered identity must be in this wallet.

**Audit logging:** Yes.

## 4. `takeoffer`

**Description:**
Accept an existing on-chain offer. If the wallet can afford the swap, this creates and posts a transaction that atomically executes the exchange — both sides swap in a single transaction, or neither does.

When accepting an offer that involves an identity (in `deliver` or `accept`), always include `parent` in the identity definition to ensure correct namespace resolution, especially for sub-identities.

**ID Control Tokens:** If the identity being traded has a control token, use the control token currency in `deliver`/`accept` instead of the identity name. Deliver: `{"currency": "idname.parent", "amount": 0.00000001}`. Accept: `{"address": "destinationaddress", "currency": "idname.parent", "amount": 0.00000001}`. See `makeoffer` for details on how to detect control tokens.

**Disabled in read-only mode.**

**Input Schema:**

| Param | Type | Required | Description |
|---|---|---|---|
| `chain` | string | Yes | Chain to transact on (e.g., `"VRSC"`, `"vrsctest"`) |
| `fromaddress` | string | Yes | VerusID, Sapling address, or wildcard address to send funds/fees from. `"*"`, `"R*"`, or `"i*"` are valid wildcards. |
| `txid` | string | Yes* | Transaction ID of the offer to accept (from `getoffers` or `makeoffer` output). *Either `txid` or `tx` is required. |
| `tx` | string | Yes* | Hex transaction to complete (from `makeoffer` with `returntx=true`). *Either `txid` or `tx` is required. |
| `changeaddress` | string | No | Transparent address or VerusID for change. |
| `deliver` | object/string | Yes | What the taker delivers to the maker — `"idname@"` (identity) or `{"currency": "name", "amount": n}` (currency). |
| `accept` | object | Yes | What the taker receives — `{"address": "addr", "currency": "name", "amount": n}` (currency) or full identity definition (identity). |
| `returntx` | boolean | No | If `true`, return signed transaction hex instead of broadcasting. Default: `false`. |
| `feeamount` | number | No | Custom fee amount. Default: `0.0001`. |

**`deliver` — what the taker gives:**
- For currency: `{"currency": "currencynameorid", "amount": n}`
- For identity: `"fullidname@"` (string shorthand)

**`accept` — what the taker receives:**
- For currency: `{"address": "destinationaddress", "currency": "currencynameorid", "amount": n}`
- For identity: full identity definition with `name`, `parent`, `primaryaddresses`, `minimumsignatures`, etc. (same structure as `registeridentity` / `updateidentity`). Always include `parent`.

**Annotations:**
```json
{
  "readOnlyHint": false,
  "destructiveHint": false,
  "idempotentHint": false,
  "openWorldHint": true,
  "confirmationHint": true
}
```

**Output:**

| Field | Type | Description |
|---|---|---|
| `txid` | string | Transaction ID of the completed swap (if `returntx` is false) |
| `hextx` | string | Serialized transaction hex (if `returntx` is true) |

**Key fields for agents:**
- Before taking an offer, verify it hasn't expired by checking `blockexpiry` from `getoffers` against the current block height.
- `deliver` is what you give, `accept` is what you get — don't mix these up.
- When delivering an identity, the identity must be in this wallet.
- When accepting an identity, the identity definition in `accept` determines the new state of the received identity (new primary addresses, authorities, etc.).

**Audit logging:** Yes.

## 5. `closeoffers`

**Description:**
Close (cancel) open offers and reclaim the locked funds. Always closes expired offers automatically, even if no parameters are given. When specific offer txids are provided, closes only those offers.

Use this to cancel offers that are no longer wanted, or as periodic cleanup to reclaim funds from expired offers.

**Disabled in read-only mode.**

**Input Schema:**

| Param | Type | Required | Description |
|---|---|---|---|
| `chain` | string | Yes | Chain to transact on (e.g., `"VRSC"`, `"vrsctest"`) |
| `txids` | string[] | No | Array of offer transaction IDs to close (from `listopenoffers` or `makeoffer` output). If omitted, closes all expired offers. |
| `destination` | string | No | Transparent or private address to send reclaimed funds to. |
| `privatefundsdestination` | string | No | Private (Sapling) address for native funds only. |

**Annotations:**
```json
{
  "readOnlyHint": false,
  "destructiveHint": false,
  "idempotentHint": false,
  "openWorldHint": true,
  "confirmationHint": true
}
```

**Output:**

Returns null on success.

**Key fields for agents:**
- Call with no `txids` to clean up all expired offers in the wallet.
- Call with specific `txids` to cancel active (unexpired) offers.
- Reclaimed funds go to the specified destination, or back to the original source if no destination is given.

**Audit logging:** Yes.

## Workflows

### Sell an Identity for Currency

1. **Check ownership:** `getidentity` (identity-mcp) — confirm `canspendfor: true`
2. **Check market:** `getoffers` with the identity name, `iscurrency=false` — see existing offers
3. **Create offer:** `makeoffer` with `offer: {"identity": "myid@"}`, `for: {"currency": "VRSC", "amount": 100}`, set `expiryheight` appropriately
4. **Monitor:** `listopenoffers` — check if the offer has been taken
5. **Cancel if needed:** `closeoffers` with the offer's txid

### Buy an Identity with Currency

1. **Find offers:** `getoffers` with the identity name or currency — find identities for sale
2. **Take offer:** `takeoffer` with the offer's `txid`, `deliver: {"currency": "VRSC", "amount": 100}`, `accept: {full identity definition with parent, primaryaddresses set to buyer's addresses}`
3. **Verify:** `getidentity` (identity-mcp) — confirm transfer completed

### Swap Two Identities

1. **Check ownership:** `getidentity` for the identity being offered — confirm `canspendfor: true`
2. **Create offer:** `makeoffer` with `offer: {"identity": "myid@"}`, `for: {full identity definition of desired ID, including parent}`
3. **Or take existing:** `getoffers` with the desired identity, find ID-for-ID offers, then `takeoffer`

### Currency-for-Currency Trade

1. **Create offer:** `makeoffer` with `offer: {"currency": "BTC", "amount": 1}`, `for: {"currency": "VRSC", "amount": 5000}`
2. **Or browse and take:** `getoffers` with `iscurrency=true`, then `takeoffer`

### Cleanup Expired Offers

1. **List expired:** `listopenoffers` with `unexpired=false, expired=true`
2. **Close all expired:** `closeoffers` with no txids — reclaims all locked funds from expired offers
